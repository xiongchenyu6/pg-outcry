# 安全策略

## 状态

pg-outcry 是**开源参考软件**。其核心流程（撮合、复式记账结算、预留、RLS 隔离、对账、钱包审批）已由
自动化端到端测试套件和对账不变量检查覆盖。**它尚未经过独立的第三方安全或财务审计。**在未经你自己的
审计、加固以及法律/合规审查之前，请勿用它托管真实的用户资金。

## 报告漏洞

请**私下**报告安全问题——不要公开提交 issue。

- 在本仓库使用 GitHub **Security Advisories**（“Report a vulnerability”），或
- 通过电子邮件联系维护者（参见 GitHub 个人主页）。

我们力争在几天内予以确认。请附上复现步骤及影响说明。

## 运营者加固清单

如果你要部署本项目，请至少做到：
- 上生产前关闭测试开放后台模式。当前托管测试版会给每个已登录的 Supabase Auth 用户完整后台权限，
  方便评审人员试用管理台。
- 将 `service_role` 密钥仅保留在服务端；切勿将其下发到浏览器。后台管理控制台使用 Supabase Auth
  加数据库 RBAC（`admin_operator_role` / `admin_role_permission`）；`service_role` 只用于可信服务端、CI、
  脚本或首次授权流程。
- 在 API 前面部署 TLS、速率限制和 WAF。
- **2FA 委托给 OAuth2 提供方**（GitHub/Google 在其登录处强制自家 2FA），因此 pg-outcry 不自带独立 TOTP 系统。
  若要*强制* 2FA，把登录限制为仅 OAuth（禁用邮箱/密码注册），让每个账户都经由强制 2FA 的身份提供方；运营者账户务必开启。
- 针对你的 schema 审查并收紧每一条 RLS 策略以及 `9900_lockdown` 函数白名单。
- 配置真实的 OAuth 提供方 + 电子邮件；如果你要求 KYC 优先的注册流程，请禁用开放注册。
- 设置按角色的 `statement_timeout`、连接数限制、备份/PITR 以及监控。
- 仅在人工/多方审批之后，才将提现接入真实托管；保持审计日志不可变更。
