# 基准测试

可复现：**[`scripts/bench.sh`](./scripts/bench.sh)** · [← README](./README.zh-CN.md)

</div>

> **这里所说的"一笔撮合"指什么。** 本基准测试中的每一笔撮合都是一笔*完整、持久、复式记账已结算*
> 的交易——吃单订单被撮合**并且**两侧的账本都已更新
> （约 8 次 insert + 4 次 update 以及若干次查询，提交至 WAL）。这与一个
> 报告"100 万订单簿操作/秒"的内存级 HFT 引擎**不可**相提并论；那些是非持久的订单簿变更，
> 而非已结算的交易。pg-outcry 用原始速度换取**每一笔成交上的 ACID 正确性**。

## 运行环境

| | |
|---|---|
| 主机 | 16 vCPU · 27 GiB 内存（开发者机器） |
| PostgreSQL | 17.6，**接近默认的配置**（`shared_buffers=128MB`、`synchronous_commit=on`、`wal_compression=off`） |
| `banker_round` | PL/pgSQL（原生实现；原生 C 替代版在这些数据中*未启用*） |
| 构建配置 | **基线（baseline）**——未应用任何自托管性能调优项 |

下方的数字是**在这台机器上、采用未调优配置时的参考值**——它们是下限，而非上限。请使用 `SERVICE=<key> ./scripts/bench.sh` 在你自己的硬件上复现/运行。

## 测试结果

| 指标 | 结果 |
|---|---|
| **顺序吞吐量**（1 个连接，1 个交易对） | **约 200–270 笔已撮合+已结算交易/秒** |
| 每笔撮合的引擎延迟（服务端） | **p50 ≈ 3.5 ms · p95 ≈ 6 ms · p99 ≈ 7–11 ms** |
| 经 PostgREST/HTTP 的端到端订单延迟 | **p50 ≈ 9 ms · p95 ≈ 22 ms · p99 ≈ 66 ms** |
| **并发扩展性**（6 个交易对并行） | **合计约 560–730 笔交易/秒**（约为单交易对的 2.5–3.7×） |

### 解读结果
- **按交易对的并发是有效的。** 由于撮合是*按品种*用一把咨询锁（advisory lock）串行化的，独立的交易对可以并行运行——合计吞吐量随交易对数量上升（6 个交易对 ≈ 单个的 3 倍）。一个拥有数十个交易对的真实交易场所会进一步扩展，直至受限于 WAL/IO。
- **单交易对延迟是毫秒级且持久的。** 一笔完整结算的交易 p50 约 3.5 ms，每一笔成交都经过 ACID 提交。这足以从容覆盖零售/区域性/山寨币交易场所；它**不是**同机房部署的微秒级 HFT 引擎（见 [WHY.md](./WHY.zh-CN.md) 第 9 节）。

## 余量——性能配置能带来什么

这些数字使用的是**基线**配置。自托管高性能配置与调优会大幅提升上限：

- `synchronous_commit = off`——对写密集型结算而言这是最大的单一杠杆（代价是崩溃时丢失最后几笔已提交的事务）。`scripts/perf-tune-local.sh RISKY=1`。
- 原生 **C `banker_round`** 替代版（在这个热点辅助函数上约 2.8×）——`ext/oc_fastmath`。
- 更大的 `shared_buffers` / `max_wal_size`，`wal_compression=on`。
- **UNLOGGED** 内存级订单簿（已包含在迁移中）——活跃订单簿无 WAL。
- 横向扩展：**按交易对分片**到多个节点（一个 CEX 没有跨交易对的事务）——随分片数量近似线性扩展。

> 结论：一个**单一、未调优的 PostgreSQL** 已经能以毫秒级延迟服务每秒数百笔完整结算的交易，并随交易对数量扩展——这对本项目所面向的中小型交易场所而言已绰绰有余，且有一条清晰、有文档记录的进一步提升路径。
